# Think Mode - 深度分析模式

> **先思考，后行动。想清楚，再动手。**

在动手写代码之前，进行系统化的分析和推理。本技能专注于**纯思考分析**，如需查阅代码或资料，应在分析完成后进入执行阶段。

## 核心原则

### 三不原则
1. **不动手** - 分析阶段禁止生成业务代码、禁止修改文件
2. **不急躁** - 禁止在理解问题前就给出方案，先复述再分析
3. **不单一** - 禁止只给出一个假设或方案，至少提供 2-3 个可能性

### 三要原则
1. **要深度** - 每个假设/方案需追溯至根本原因/核心思路
2. **要量化** - 用置信度% 表达不确定性，说明判断依据
3. **要收敛** - 分析最终要收敛到可执行的验证步骤或明确建议

## 触发场景

| 场景类型 | 典型问题 | 本技能作用 |
|----------|----------|------------|
| **复杂 Bug 排查** | 偶发问题、报错无日志、性能问题 | 生成假设树 + 验证路径 |
| **方案设计讨论** | 架构选型、功能设计、技术方案 | 多方案对比 + 推荐理由 |
| **技术咨询建议** | 技术选型、好不好用、要不要做 | 利弊分析 + 风险评估 |
| **代码理解分析** | 这段代码什么问题、为什么这样写 | 逐层剖析 + 意图推测 |

## 工作流程

### 第一步：复述与澄清（必选）

用你自己的话复述问题，确认理解一致。如发现模糊点，立即使用 `AskUserQuestion` 提问。

**输出格式：**
```
## 问题理解
[用自己的话复述，而不是重复用户的原话]

## 需要澄清的问题（如有）
1. [问题 1] - 影响后续分析方向
2. [问题 2] - 影响技术方案选择
```

### 第二步：模式识别与分析

根据问题特征选择分析模式。**注意：此阶段仅输出分析，不执行任何工具调用。**

---

### 模式 A：排查模式（Debug Analysis）

**触发信号**：bug、报错、为什么、排查、不正常、没反应、失败了

**分析框架：**
```markdown
## 可能原因假设（按优先级排序）

### 假设 1：[核心概括]（置信度：高/中/低 ≈ XX%）
- **推测根因**：[深入到机制层面，不是表面描述]
- **支持证据**：[当前有哪些现象支持这个假设]
- **证伪方法**：[如何排除这个可能性]
- **验证命令/步骤**：[具体可执行]

### 假设 2：[核心概括]（置信度：≈ XX%）
...

## 排查路径建议
```
**优先级排序理由**：
- 高置信度：有明显症状支持，或历史经验高频
- 中置信度：逻辑上可能，但缺乏直接证据
- 低置信度：理论上可能，但概率较低

---

### 模式 B：设计模式（Design Analysis）

**触发信号**：实现、功能、怎么设计、方案、架构、怎么做

**分析框架：**
```markdown
## 技术方案对比

### 方案 A：[方案名称]（推荐指数：★★★☆☆）
- **核心思路**：[一句话概括本质]
- **技术选型**：[具体技术栈/工具]
- **优点**：[2-3 条，说清为什么是优点]
- **缺点/风险**：[2-3 条，不说风险的建议不可信]
- **适用场景**：[什么情况下选这个]

### 方案 B：[方案名称]（推荐指数：★★★★☆）
...

## 推荐方案及理由
**推荐**：方案 [X]

**核心理由**：
1. [理由 1] - [支撑论据]
2. [理由 2] - [支撑论据]

**不推荐其他方案的原因**：
- 方案 [Y]：[具体原因]

## 架构设计要点（如适用）
- 模块划分：...
- 数据流向：...
- 关键接口：...
```

---

### 模式 C：咨询模式（Consulting Analysis）

**触发信号**：推荐、选哪个、好不好、值不值得、要不要

**分析框架：**
```markdown
## 对比分析

| 维度 | 选项 A | 选项 B | 胜出方 |
|------|--------|--------|--------|
| 学习成本 | ... | ... | A/B |
| 开发效率 | ... | ... | A/B |
| 社区生态 | ... | ... | A/B |
| 长期维护 | ... | ... | A/B |

## 核心差异
- **选项 A 的本质**：[深层特点]
- **选项 B 的本质**：[深层特点]

## 建议
**推荐**：[明确选项]

**理由**：
1. [理由 1]
2. [理由 2]

**前提假设**：[你的建议在什么条件下成立]

## 风险提示
- [风险 1]：[可能的后果 + 缓解措施]
- [风险 2]：...
```

---

### 模式 D：代码理解模式（Code Analysis）

**触发信号**：这段代码、为什么这样写、什么问题、怎么理解

**分析框架：**
```markdown
## 代码意图分析

### 表层功能
[这段代码在做什么]

### 深层设计
- **为什么这样组织**：[结构背后的考量]
- **潜在问题**：[可能的 bug 或坏味道]
- **约束条件**：[当时的设计限制]

### 可能的改进方向
- [方向 1]：[改进思路]
- [方向 2]：[改进思路]
```

---

## 输出质量检查清单

在输出分析前，自检以下问题：

- [ ] 我是否真正理解了问题？（能用自己的话复述）
- [ ] 是否提供了至少 2 个假设/方案？
- [ ] 每个假设/方案是否有置信度/推荐指数？
- [ ] 是否说明了判断依据，而非拍脑袋？
- [ ] 是否指出了风险/缺点，而非只说好处？
- [ ] 分析是否可执行、可验证？

## 行为约束

### ❌ 禁止事项
- 在分析阶段生成业务代码
- 未经澄清就直接给出方案
- 只给出单一假设或方案
- 用"可能"、"也许"模糊带过，不给置信度
- 只说优点，不谈风险

### ✅ 强制事项
- 必须先复述问题
- 必须表达不确定性（置信度/推荐指数）
- 必须说明判断依据
- 必须等待用户确认后再进入执行

## 下一步引导

分析完成后，明确告知用户如何继续：

```
---
**下一步建议**：
- 如需深入验证某个假设，回复"继续验证假设 X"
- 如需开始实施方案，回复"按方案 X 执行"
- 如有其他疑问，请直接提出
```

**示例 1：复杂 Bug 排查**

```
用户：/think 线上 API 偶尔返回 500，日志没报错，帮我分析下

Claude：
## 问题理解
线上 API 出现间歇性 500 错误，但应用日志中没有对应的错误堆栈，需要定位根因。

需要确认几个关键信息：
1. 频率大概是多少？（每天几次还是每小时几次？）
2. 是固定某个接口还是多个接口都会出现？
3. 500 错误返回有响应体内容吗？还是完全空白？

用户：每天 3-5 次，主要是登录接口，返回内容也是 500 的标准错误页

Claude：
## 可能原因假设（按优先级排序）

### 假设 1：数据库连接池耗尽（置信度：高 ≈ 60%）
- **推测根因**：登录高峰时连接池被占满，新请求等待超时，但超时异常可能未被正确捕获记录
- **支持证据**：登录接口高频出现、偶发性特征符合资源耗尽特点
- **证伪方法**：检查数据库连接池监控指标，看是否有"等待获取连接超时"的计数
- **验证命令**：查看数据库中间件监控或日志中是否有连接超时相关指标

### 假设 2：上游认证服务超时（置信度：中 ≈ 30%）
- **推测根因**：登录依赖的 OAuth/SSO 服务偶尔响应慢，导致整个请求超时
- **支持证据**：登录接口依赖外部服务，外部服务不稳定是常见问题
- **证伪方法**：检查链路追踪系统，看 500 错误发生时上游服务响应时间是否异常
- **验证步骤**：如有 SkyWalking/Zipkin 等，过滤登录接口的慢调用链

### 假设 3：进程被系统 OOM Killer 杀死（置信度：低 ≈ 10%）
- **推测根因**：内存不足时进程被内核杀死，来不及写应用日志
- **支持证据**：能解释为什么应用日志无记录
- **证伪方法**：检查系统日志 `dmesg | grep -i "out of memory"` 或 `/var/log/messages`

## 排查路径建议
1. **优先验证假设 1**：连接池问题最常见，检查成本也最低
2. **同时检查假设 2**：如有链路追踪系统，可快速验证
3. **最后排查假设 3**：需要查看系统层面日志

---
**下一步建议**：
- 如需我帮你分析具体验证步骤，回复"继续验证假设 1"
- 如已有监控数据需要解读，可以直接发给我
```

**示例 2：技术选型咨询**

```
用户：/think 新项目选 Next.js 还是 Nuxt，团队主要写 Vue

Claude：
## 问题理解
团队技术栈以 Vue 为主，需要为新技术栈选型，在 Next.js(React) 和 Nuxt(Vue) 之间做决策。

## 对比分析

| 维度 | Next.js (React) | Nuxt (Vue) | 胜出方 |
|------|-----------------|------------|--------|
| 团队上手成本 | 需学习 React 思维 | 无需额外学习 | Nuxt |
| 招聘难度 | React 开发者多 | Vue 开发者也多 | 平手 |
| 生态丰富度 | 更成熟，组件更多 | 足够用 | Next.js |
| Vercel 部署 | 原生支持 | 支持 | Next.js |
| 学习曲线 | 较陡（JSX、Hooks）| 较平缓 | Nuxt |
| 长期趋势 | Meta 背书，AI 友好 | 社区稳定 | Next.js |

## 核心差异的本质
- **Next.js**：选择它意味着拥抱 React 生态，换取更丰富的第三方库和 AI 工具支持
- **Nuxt**：选择它意味着保持团队现有技能，降低学习成本

## 建议
**推荐**：取决于团队优先级

### 如果优先考虑「快速上手、快速交付」→ Nuxt 3
**理由**：团队无需学习新框架，Vue 的经验可直接复用

### 如果优先考虑「长期发展、生态广度」→ Next.js 14
**理由**：
1. React 生态更成熟，遇到问题更容易找到解决方案
2. 与 AI 工具链集成更好（Claude Code、Cursor 等对 React 支持更优）
3. 招聘市场 React 开发者占比更高

## 风险提示
- **选 Nuxt 的风险**：Vue 在国内虽流行，但全球趋势偏向 React，长期可能面临生态边缘化
- **选 Next.js 的风险**：团队需要 1-2 个月适应期，初期开发效率会下降

---
**下一步建议**：
- 如需我评估具体迁移成本，回复"评估 React 迁移成本"
- 如需详细的 Next.js 学习路线，回复"给出 React 学习计划"
```

**示例 3：方案设计讨论**

```
用户：/think 想做个文件上传功能，支持断点续传，怎么设计比较好

Claude：
## 问题理解
需要设计一个支持断点续传的文件上传系统，需要技术方案设计。

需要澄清：
1. 预计文件大小范围？（MB 级还是 GB 级）
2. 多端上传需求？（Web、App 都需要吗）
3. 是否需要秒传功能？

用户：主要是 Web 端，文件 100MB-2GB 都有，秒传暂时不需要

Claude：
## 技术方案对比

### 方案 A：分片上传 + 服务端合并（推荐指数：★★★★★）
- **核心思路**：前端切片→并发上传→服务端暂存→全部完成后合并
- **技术选型**：
  - 前端：原生 Blob.slice 或 SparkMD5 计算分片 hash
  - 后端：Go/Node.js 适合处理高并发分片写入
- **优点**：
  - 断点续传自然支持（检查已上传分片）
  - 并发上传提升速度
  - 失败重试成本低（仅重传失败分片）
- **缺点**：
  - 服务端需要管理临时分片文件
  - 合并大文件时占用 IO
- **适用场景**：大文件、网络不稳定环境

### 方案 B：Range 请求 + 后端断点记录（推荐指数：★★★☆☆）
- **核心思路**：利用 HTTP Range 头，后端记录已接收字节位置
- **技术选型**：需要支持 Range 的 Web 服务器
- **优点**：
  - 协议层面原生支持
  - 实现相对简单
- **缺点**：
  - 对服务器配置有要求
  - 并发性能不如分片方案
- **适用场景**：中等文件大小，对并发要求不高

## 推荐方案：方案 A（分片上传）

**核心理由**：
1. **更适合大文件**：2GB 文件分片后可并发上传，体验远优于单请求
2. **断点续传更精确**：以分片为单位检查，比字节级更可靠
3. **失败恢复成本更低**：只需重传失败的分片，不是从断点重新开始

## 架构设计要点

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   前端      │────▶│   API GW    │────▶│ 上传服务    │
│  分片 + 并发 │     │  限流鉴权   │     │ 分片存储    │
└─────────────┘     └─────────────┘     └─────────────┘
                                              │
                                              ▼
                                       ┌─────────────┐
                                       │   临时目录   │
                                       │  分片暂存   │
                                       └─────────────┘
                                              │
                                              ▼ (全部完成后)
                                       ┌─────────────┐
                                       │   对象存储   │
                                       │  合并上传   │
                                       └─────────────┘
```

**关键接口设计**：
1. `POST /upload/init` - 初始化上传，获取 uploadId
2. `POST /upload/chunk` - 上传分片（可并发）
3. `POST /upload/complete` - 合并分片，完成上传
4. `GET /upload/status?uploadId=xxx` - 查询已上传分片（用于续传）

---
**下一步建议**：
- 如需我给出详细 API 定义，回复"设计 API"
- 如需前端分片上传代码示例，回复"前端实现"
```
