# 项目文档自动更新

根据项目的代码改动，自动识别并更新所有相关文档，确保文档与代码实现保持同步。

## 文档类型识别

### 1. 需求文档 (PRD)
- **位置**：`docs/产品需求文档(PRD).md`
- **更新场景**：
  - 新增功能或模块
  - 功能变更或移除
  - 用户交互流程变化
  - 新增配置项或参数

### 2. 技术设计文档 (TDD)
- **位置**：`docs/技术设计文档(TDD).md`
- **更新场景**：
  - 新增模块、类、函数
  - 架构调整
  - 技术栈变更
  - 接口设计变更

### 3. API接口文档
- **位置**：`docs/API接口文档.md`
- **更新场景**：
  - 新增API端点
  - API参数变更
  - 返回值结构变化
  - 接口废弃或新增

### 4. 数据字典
- **位置**：`docs/数据字典.md`
- **更新场景**：
  - 数据库表结构变更
  - 新增字段或表
  - 字段类型或约束变化
  - 数据模型调整

### 5. 文档索引
- **位置**：`docs/文档索引.md`
- **更新场景**：
  - 版本号变更
  - 新增或删除文档
  - 目录结构调整

## 处理流程

### 1. 分析项目改动

```bash
# 查看最近的 git 历史
git log -10 --oneline

# 查看未提交的改动
git status
git diff

# 查看已提交但未推送的改动
git log origin/main..HEAD --oneline
```

### 2. 识别改动的性质和范围

分析以下信息：
- **改动类型**：新功能、bug修复、重构、配置调整等
- **影响范围**：哪些模块、哪些文件
- **变更层级**：
  - 用户可见功能 → 需要 PRD 更新
  - 代码结构变更 → 需要 TDD 更新
  - API 接口变更 → 需要 API 文档更新
  - 数据模型变更 → 需要数据字典更新

### 3. 检查现有文档

```bash
# 列出所有文档文件
ls -la docs/*.md discuss/*.md
```

对于每个文档：
- 读取文档内容
- 理解文档结构和章节划分
- 识别需要更新的章节

### 4. 更新文档

根据改动类型和影响范围，更新相应的文档章节：

#### 更新 PRD
```markdown
## 功能变更记录

### [日期] <功能名称>
- **变更类型**：新增/修改/移除
- **影响范围**：<受影响的模块>
- **变更内容**：
  - <具体变更点1>
  - <具体变更点2>
- **用户体验变化**：<描述用户可见的变化>
```

#### 更新 TDD
```markdown
## 技术实现更新

### <模块名称> - [日期]
- **变更内容**：<实现细节>
- **技术方案**：<设计思路>
- **代码位置**：<相关文件路径>
- **依赖关系**：<依赖的模块或服务>
```

#### 更新 API 文档
```markdown
## API 变更记录

### <接口名称> - [日期]
- **变更类型**：新增/修改/废弃
- **请求路径**：`<HTTP 方法> <路径>`
- **变更内容**：
  - 新增参数：`<参数名>` - <说明>
  - 修改参数：`<参数名>` - <变更说明>
  - 返回值变化：<描述>
```

#### 更新数据字典
```markdown
## 数据模型变更

### <表名或模型名> - [日期]
- **变更类型**：新增/修改/删除
- **字段变更**：
  - 新增字段：`<字段名>` <类型> <说明>
  - 修改字段：`<字段名>` <原类型> → <新类型> <说明>
  - 删除字段：`<字段名>` <原因>
```

#### 更新文档索引
```markdown
## 版本历史

| 版本号 | 日期 | 主要变更 |
|--------|------|----------|
| vX.X.X | YYYY-MM-DD | <变更摘要> |
```

### 5. 验证更新质量

- [ ] 所有相关文档都已更新
- [ ] 文档内容准确反映代码实现
- [ ] 日期和版本号正确
- [ ] 没有遗漏的重要变更
- [ ] 文档格式和风格一致

### 6. 提交文档更新

```bash
# 添加更新的文档
git add docs/*.md discuss/*.md

# 提交
git commit -m "$(cat <<'EOF'
docs: 根据代码实现更新项目文档

- 更新产品需求文档(PRD)
- 更新技术设计文档(TDD)
- 更新API接口文档
- 更新数据字典
- 更新文档索引版本号

🤖 Generated with [Claude Code](https://claude.com/claude-code)

Co-Authored-By: Claude Sonnet 4.5 <noreply@anthropic.com>
EOF
)"
```

## 文档更新原则

### 1. 及时性
- 代码改动后立即更新文档
- 不要累积多次改动再批量更新

### 2. 准确性
- 文档必须真实反映代码实现
- 避免文档与代码不一致

### 3. 完整性
- 更新所有受影响的文档
- 不要遗漏任何相关章节

### 4. 一致性
- 保持文档风格统一
- 使用一致的术语和格式

### 5. 可追溯性
- 记录变更日期和内容
- 便于后续回溯和审计

## 常见场景示例

### 场景 1：新增 API 端点

**改动**：新增 `GET /api/comments/hot` 接口

**需要更新的文档**：
1. ✅ API接口文档.md - 新增接口说明
2. ✅ 数据字典.md - 如果涉及新的数据结构
3. ✅ 产品需求文档.md - 如果是用户可见的新功能
4. ✅ 文档索引.md - 更新版本号

### 场景 2：修改数据模型

**改动**：为 Comment 表添加 `is_hot` 字段

**需要更新的文档**：
1. ✅ 数据字典.md - 更新表结构
2. ✅ API接口文档.md - 如果API返回值包含该字段
3. ✅ 技术设计文档.md - 更新数据模型设计

### 场景 3：重构代码结构

**改动**：提取评论服务为独立模块

**需要更新的文档**：
1. ✅ 技术设计文档.md - 更新架构设计
2. ✅ API接口文档.md - 如果接口路径变化

### 场景 4：Bug 修复

**改动**：修复窗口位置偏移问题

**需要更新的文档**：
1. ⚠️ 通常不需要更新文档
2. 特殊情况下，更新 TDD 记录修复方案

## 注意事项

### 文档结构
- 保持原有文档的章节结构
- 在相应章节添加或修改内容
- 不要随意删除或重组章节

### 版本号
- 文档索引中的版本号应与项目版本一致
- 遵循语义化版本规范（Semantic Versioning）

### 变更记录
- 在每个文档的顶部维护变更记录
- 格式：`[日期] 版本号 - 变更摘要`

### 冲突处理
- 如果文档已过时，先标记为 `过时内容`
- 然后添加新的正确内容
- 最后删除过时内容

### 敏感信息
- 不要在文档中暴露密钥、密码等敏感信息
- API 文档中不要包含内网地址或测试账号

## 使用示例

当用户说：
- "更新文档"
- "根据代码改动更新文档"
- "同步文档和代码"
- "update-doc"

你会：
1. 执行上述处理流程
2. 分析代码改动的性质和范围
3. 识别需要更新的文档
4. 更新所有相关文档
5. 提交文档更新

## 错误处理

### 文档不存在
```
⚠️ 警告：未找到 docs/目录
请确认项目文档结构是否符合标准
```

### 文档格式错误
```
❌ 错误：文档格式不符合要求
文件：docs/xxx.md
错误：缺少必需的章节
```

### 改动过大
```
⚠️ 警告：检测到大量改动
建议分批更新文档，避免一次性修改过多内容
```

### 无法自动识别
```
⚠️ 提示：部分改动无法自动识别对应的文档章节
需要手动确认以下内容的更新：
- <具体内容>
```
