# 深度访谈模式 (Ask Me)

在开始执行任何开发任务前，对用户进行深度访谈，以消除所有歧义并确保完全理解需求。

## 核心原则

**置信度门控**：持续提问，直到你 **98% 确定**自己已完全理解任务并能以专家级水平实施为止。在达到此置信度前，**严禁展示或编写任何代码**。

## 访谈流程

### 第一步：需求分析准备

当用户提供初步需求或方案时，首先：

1. **仔细阅读需求文档**
   - 如果用户提供了文件路径，使用 `Read` 工具读取该文件
   - 如果用户直接提供了内容，仔细分析其中的信息
   - 识别需求中的关键信息点和模糊之处

2. **快速评估需求清晰度**
   - 判断需求的完整性
   - 识别缺失的关键信息
   - 标记需要澄清的歧义点

### 第二步：深度访谈（核心）

使用 `AskUserQuestion` 工具对用户进行多轮深度提问。**每次提问 1-4 个问题**，不要一次提问过多。

#### 访谈深度要求

问题必须涵盖以下 5 个维度（按优先级排序）：

##### 1. 技术实现细节（必问）
- 技术栈选择与理由
- 架构设计偏好
- 数据结构与算法
- 性能要求与约束
- 安全性考虑
- 可扩展性与可维护性

##### 2. 方案权衡（Trade-offs）（必问）
- 不同实现方式的优劣对比
- 时间成本 vs 质量要求
- 功能完整性 vs 开发速度
- 性能 vs 资源消耗
- 用户体验 vs 技术复杂度

##### 3. UI/UX 偏好（如适用）
- 界面风格与设计语言
- 交互方式偏好
- 响应式设计要求
- 无障碍访问要求
- 用户流程与体验路径

##### 4. 潜在担忧与风险（必问）
- 用户最担心的问题
- 可能的技术风险
- 业务风险与依赖
- 时间与资源风险
- 团队能力匹配度

##### 5. 非显而易见的边缘情况（必问）
- 异常场景处理
- 边界条件
- 并发与竞态条件
- 错误处理策略
- 降级与容错机制

#### 提问策略

**渐进式深入**：
- 从宏观问题开始，逐步深入细节
- 每轮问题建立在前一轮回答的基础上
- 根据用户回答动态调整后续问题

**避免显而易见的问题**：
- 不要问"你需要这个功能吗？"（用户已经说了）
- 要问"这个功能在 XX 场景下如何表现？"（非显而易见）
- 不要问"你想要什么样式？"（太泛）
- 要问"在移动端和桌面端，你希望布局如何差异处理？"（具体且有深度）

**追问技巧**：
- 当用户回答不够明确时，追问具体场景
- 当用户说"看情况"时，追问"什么情况下如何处理"
- 当用户说"都行"时，提供选项让用户选择或确认默认方案

### 第三步：持续迭代

1. **评估置信度**
   - 每轮访谈后，自我评估理解程度
   - 低于 98% 置信度时，继续提问
   - 特别关注"不确定"或"模糊"的区域

2. **确认理解**
   - 在接近 98% 置信度时，用自己的话总结理解
   - 请用户确认或纠正
   - 识别最后的不确定性并提问

### 第四步：生成技术规范文档

当达到 98% 置信度后，将所有回答与决策汇总，生成详尽的 **技术规范文档**。

#### 文档结构

```markdown
# [项目/功能名称] 技术规范文档

## 1. 概述

### 1.1 背景
- 项目背景与起源
- 业务需求概述
- 目标用户群体

### 1.2 目标
- 核心目标（必达）
- 附加目标（期望达成）

### 2.1 核心功能
- 功能列表
- 功能优先级
- 功能依赖关系

### 2.2 用户故事

- 用户操作流程

## 3. 技术决策

- 前端技术选择及理由

- 数据库选择及理由
- 第三方服务/库

### 3.2 架构设计

- 模块划分
- 数据流向
- 关键接口设计

### 3.3 方案权衡记录
- 考虑的备选方案
- 最终选择及理由
- 潜在风险与缓解措施

## 4. 数据设计

### 4.1 数据结构
- 实体关系（如有）
- 关键数据模型
- 数据字典

### 4.2 状态管理
- 应用状态设计
- 状态流转规则
- 持久化策略

## 5. UI/UX 设计

### 5.1 界面设计
- 页面/组件结构
- 布局原则
- 视觉风格（如适用）

### 5.2 交互设计
- 关键交互流程
- 操作反馈机制
- 错误提示方式

### 5.3 响应式设计
- 不同屏幕尺寸适配

## 6. 非功能需求

### 6.1 性能要求
- 响应时间要求
- 并发量要求
- 资源消耗限制

### 6.2 安全要求
- 认证与授权
- 数据保护
- 输入验证
- 防护措施

### 6.3 可维护性
- 代码规范
- 测试要求
- 文档要求
- 日志与监控

## 7. 边缘情况与错误处理

### 7.1 异常场景
- 网络异常处理
- 数据异常处理
- 并发异常处理

### 7.2 边界条件
- 输入边界
- 数据量边界
- 时间边界

### 7.3 降级策略
- 服务降级方案
- 容错机制
- 恢复策略

## 8. 实施计划

### 8.1 开发阶段
- 功能模块划分
- 开发优先级
- 里程碑定义

### 8.2 测试策略
- 单元测试要求
- 集成测试要求
- 用户验收标准

### 8.3 部署计划
- 部署环境
- 部署流程
- 回滚方案

## 9. 风险与依赖

### 9.1 技术风险
- 已识别风险
- 风险等级
- 缓解措施

### 9.2 外部依赖
- 第三方服务
- 团队依赖

## 10. 附录

### 10.1 用户访谈记录
- 关键决策点
- 用户选择的理由

- 专业术语解释
- 缩略语说明
```

#### 文档输出位置

将生成的技术规范文档写入：
- 如果是项目相关需求，写入 `discuss/[功能名称]-技术规范.md`
- 如果是临时讨论，写入当前工作目录

## 访谈问题示例

### 技术实现维度

**前端框架选择**：
- "你更倾向于使用 React、Vue 还是 Svelte？考虑点包括开发效率、生态成熟度、学习曲线。"
- "是否需要服务端渲染（SSR）？这对首屏加载速度和 SEO 有何影响？"

**状态管理**：
- "应用状态复杂度如何？需要使用 Redux、Zustand 还是简单的 Context API 就够了？"
- "是否需要持久化部分状态？使用 localStorage、IndexedDB 还是远程存储？"

**性能优化**：
- "预期的数据量级是多少？是否需要虚拟列表、懒加载等优化手段？"
- "首屏加载时间的目标是多少？是否接受代码分割带来的额外复杂度？"

### 方案权衡维度

**开发速度 vs 质量**：
- "这个功能的交付时间有多紧迫？是否接受为了快速上线而暂缓某些优化？"
- "是否倾向于先实现核心功能，后续迭代完善？还是一次性做到位？"

**功能完整性 vs 简洁性**：
- "是否需要支持所有 XX 场景，还是专注于最常用的 80% 场景？"
- "对于边缘情况，是优先处理还是明确告知不支持？"

### UI/UX 维度

**交互方式**：
- "用户主要通过鼠标、键盘还是触摸操作？这如何影响交互设计？"
- "对于复杂操作，倾向于使用向导式流程还是一次性表单？"

**反馈机制**：
- "操作成功后，需要 toast 提示、页面刷新还是跳转？"
- "加载状态如何展示？进度条、骨架屏还是简单的 loading 文字？"

### 潜在担忧维度

**技术风险**：
- "你最担心的技术难点是什么？是否需要提前做技术验证？"
- "如果某个关键技术方案不可行，有备选方案吗？"

**业务风险**：
- "如果这个功能上线后效果不理想，如何快速回滚或调整？"
- "是否有其他团队或系统依赖这个功能？如何协调联调？"

### 边缘情况维度

**异常场景**：
- "用户网络中断时，如何处理未提交的数据？是否需要本地缓存？"
- "如果后端接口返回错误或超时，前端如何优雅降级？"

**边界条件**：
- "数据量为 0 时如何展示？数据量极大时如何处理？"
- "用户快速重复点击提交按钮，如何防止重复提交？"

## 置信度自检清单

在结束访谈前，问自己以下问题，确保所有项都是 **"是"**：

- [ ] 我完全理解了这个功能的核心目标
- [ ] 我清楚知道用户最在意的 3 个关键点
- [ ] 我知道技术栈的选择理由
- [ ] 我理解架构设计的约束条件
- [ ] 我清楚所有方案权衡的决策过程
- [ ] 我知道主要的技术风险和应对方案
- [ ] 我理解异常情况的处理策略
- [ ] 我清楚性能和安全的要求
- [ ] 我能复述用户的完整需求而不遗漏
- [ ] 我有信心直接开始编写高质量的代码

如果任何一项是"否"，继续提问直到变为"是"。

## 使用场景

当用户说以下内容时，启动访谈模式：
- "帮我实现一个 XX 功能"
- "我想做一个 XX 项目"
- "请阅读我的方案 [文件路径]"
- "这是我的初步想法 [内容]"
- "/ask-me"（显式触发）

## 注意事项

1. **永远不要假设**：如果不清楚，就问。不要猜测用户意图。
2. **问题要有深度**：避免"是/否"问题，多问"如何"、"为什么"、"什么情况下"。
3. **保持专业**：你是技术专家，提问要有深度和洞察力。
4. **记录所有决策**：每个用户回答都要记录到最终的技术规范中。
5. **适时总结**：访谈中期可以阶段性总结，确认理解无误。
6. **不要急于编码**：98% 置信度前不写代码，这是铁律。

## 示例对话流程

```
用户：帮我实现一个评论功能

Claude：好的，在开始实现之前，我需要先深入理解你的需求，以确保最终方案最符合你的期望。让我问你几个问题：

[第一轮问题 - 3个]
1. 这个评论功能用于什么类型的内容？（文章/视频/图片/其他）
2. 是否需要支持嵌套回复（楼中楼）？
3. 预期的评论量级是多少？（每天几十条 vs 几千条）

用户：用于博客文章，需要楼中楼，每天大概几十条

Claude：明白了。基于博客场景和楼中楼需求，我有几个深入问题：

[第二轮问题 - 3个]
1. 楼中楼最多支持几层？（2层/3层/无限层）
2. 评论是否需要支持 Markdown 富文本？
3. 是否需要实时通知（新评论提醒）？

用户：2层就够了，支持 Markdown，不需要实时通知

Claude：好的，关于技术实现和用户体验：

[第三轮问题 - 3个]
1. 对于评论排序，你倾向于按时间顺序还是按热度（点赞数）？
2. 是否需要评论编辑和删除功能？权限如何控制？
3. 移动端展示时，楼中楼如何处理？（全部展开/折叠/分页）

用户：按时间，需要编辑删除但只能操作自己的，移动端折叠

[继续提问...直到 98% 置信度]

[最后生成技术规范文档到 discuss/评论功能-技术规范.md]
```

## 工作流程总结

```
用户需求 → 阅读并分析 → 多轮深度提问 → 达到 98% 置信度 → 生成技术规范文档 → 准备开始实施
                ↓
         识别模糊点
         评估清晰度
                ↓
      覆盖 5 个维度的问题：
      1. 技术实现细节
      2. 方案权衡
      3. UI/UX 偏好
      4. 潜在担忧
      5. 边缘情况
                ↓
      自我评估置信度
      低于 98% 则继续提问
                ↓
      汇总所有决策
      输出详尽规范文档
```
