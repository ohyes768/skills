# 深度访谈模式 (Ask Me)

这是一个 Claude Code 自定义 skill（命令），在开始任何开发任务前，通过深度访谈确保完全理解需求，最终生成详细的技术规范文档。

## 为什么需要这个 skill？

软件开发中最大的风险之一是**理解偏差**：
- 你说 A，开发者理解成 B
- 开发者做了 C，而你期望的是 A
- 边缘情况没考虑清楚，上线后出问题
- 方案权衡没讨论，选错了技术路线

这个 skill 通过**深度访谈**，在写代码前就消除所有歧义，确保：
- ✅ 完全理解你的需求
- ✅ 考虑所有边缘情况
- ✅ 讨论清楚方案权衡
- ✅ 生成详尽的技术规范文档

## 工作原理

1. **你提供初步需求**（文字描述、方案文档、或只是个想法）
2. **Claude 深度访谈你**（多轮提问，覆盖 5 个维度）
3. **持续提问直到 98% 确定**（置信度门控）
4. **生成技术规范文档**（作为后续实施的唯一事实来源）

### 提问的 5 个维度

1. **技术实现细节** - 技术栈、架构、性能、安全
2. **方案权衡 (Trade-offs)** - 不同方案的优劣对比
3. **UI/UX 偏好** - 界面风格、交互方式（如适用）
4. **潜在担忧与风险** - 技术风险、业务风险
5. **边缘情况** - 异常场景、边界条件、错误处理

## 使用方法

### 方式 1：直接触发
在对话中说：
- "帮我实现一个 XX 功能"
- "我想做一个 XX 项目"
- "这是我的方案 [文件路径或内容]"

### 方式 2：显式调用
说：
- "/ask-me"
- "启动访谈模式"

## 示例对话

```
你：帮我实现一个评论功能

Claude：好的，在开始实现之前，我需要先深入理解你的需求...
[开始提问]

Claude：这个评论功能用于什么类型的内容？
你：博客文章

Claude：是否需要支持嵌套回复（楼中楼）？
你：需要

Claude：预期的评论量级是多少？
你：每天大概几十条

Claude：楼中楼最多支持几层？
你：2层就够了

[继续深入提问...]

[最后生成一份详尽的技术规范文档到 discuss/评论功能-技术规范.md]
```

## 生成的技术规范文档包含

- **概述** - 背景、目标、非目标
- **功能需求** - 核心功能、用户故事
- **技术决策** - 技术栈、架构、方案权衡
- **数据设计** - 数据结构、状态管理
- **UI/UX 设计** - 界面、交互、响应式
- **非功能需求** - 性能、安全、可维护性
- **边缘情况与错误处理** - 异常场景、降级策略
- **实施计划** - 开发阶段、测试、部署
- **风险与依赖** - 技术风险、外部依赖
- **附录** - 访谈记录、参考资料

## 核心特性

### 置信度门控
Claude 会持续提问，直到自己 **98% 确定**完全理解任务。在此之前，**不会编写任何代码**。

### 渐进式深入
- 从宏观问题开始
- 逐步深入细节
- 根据回答动态调整后续问题

### 避免显而易见的问题
- 不问"你需要这个功能吗？"（你已经说了）
- 问"这个功能在 XX 场景下如何表现？"（非显而易见）

### 记录所有决策
每个问题、回答、决策都会记录到最终的技术规范文档中。

## 适合的使用场景

✅ **适合**：
- 新功能开发
- 项目重构
- 技术方案选型
- 架构设计
- 需求澄清

❌ **不适合**：
- 简单的 bug 修复（一行代码那种）
- 明确的文档更新
- 配置调整
- 已有详细规范的实施（已经有了 PRD/TDD）

## 自定义

你可以编辑 `SKILL.md` 文件来自定义：
- 提问的维度和优先级
- 技术规范文档的结构
- 置信度门控的阈值
- 输出文档的位置

## 注意事项

1. **耐心回答问题** - 这些问题都是为了确保最终结果符合你的期望
2. **不要说"都行"** - 尽量给出明确的偏好，或让 Claude 提供选项供你选择
3. **可以反问** - 如果不理解 Claude 的问题，可以要求澄清或举例
4. **访谈结束后可以调整** - 生成的技术规范文档不是死的，可以在后续讨论中修改

## 为什么 98% 置信度？

因为 100% 是不可能的（总有未知因素），而 95% 太低（容易出问题）。
98% 是一个**足够确定可以开始实施，但又留有余地应对未知**的甜蜜点。

## 与其他 skill 的配合

这个 skill 可以与其他 skill 配合使用：
1. **ask-me** - 深度访谈，生成技术规范
2. **其他实施 skill** - 根据技术规范进行开发
3. **git-commit-push** - 提交代码
4. **update-doc** - 更新项目文档

## 常见问题

**Q: 访谈要多久？**
A: 取决于需求复杂度。简单功能可能 3-5 轮提问，复杂项目可能 10+ 轮。但这是值得的投资，避免后期返工。

**Q: 可以跳过某些问题吗？**
A: 可以回答"使用你的最佳判断"或"暂时不考虑，后续迭代"。

**Q: 生成的技术规范可以修改吗？**
A: 当然可以。访谈结束后，你可以直接修改生成的文档，或继续讨论调整。

**Q: 如果我改变主意了怎么办？**
A: 随时可以告诉 Claude 新的想法，会继续讨论并更新技术规范。
